# CONNECT call architecture (Call-Arch-1)

**Status:** Planning — aligns paid **voice** with existing **`POST /v2/billing/call-charge/start`** and **`POST /v2/billing/call-charge/settle`** (see **`docs/connect-call-charging.md`**). Media is still **off** until this contract is implemented end-to-end.

---

## 1. First shipped call scope (locked)

| Decision | Choice | Why |
|----------|--------|-----|
| Modality | **Voice only** | Lowest bandwidth/latency surface, one A/V pipeline to harden, simpler QA and tariff (voice rate only in v1). |
| Topology | **1:1 only** | CONNECT rooms are pairwise chat; no multiparty SFU fan-out or UI for N-way in v1. |
| Video | **Later (Phase Call-Arch-N)** | Tariff already has **`video.coinsPerSecond`** for future use; ship voice + billing correctness before turning on cameras. |

**Non-goals for v1:** PSTN dial-out, group calls, screen share, server-side recording.

---

## 2. Recommended media stack: **LiveKit**

### 2.1 Options considered

| Option | Fit for CONNECT | Notes |
|--------|------------------|--------|
| **LiveKit** (OSS SFU + Cloud) | **Strong** | WebRTC SFU, **room + participant** model maps to **`callSessionId`**; **server-minted access tokens** (no phone/email); official **React Native** SDK; can **self-host** or use LiveKit Cloud; webhook/API for participant join/leave for **duration sanity checks**. |
| **Twilio Voice / Video** | Mixed | Mature; Programmable Voice is often **PSTN-centric**; Video rooms work but pricing/model is heavier; identity often tied to Twilio identity patterns — extra care for **anonymous deviceId** mapping. |
| **Daily** | Strong | Similar to LiveKit (managed WebRTC); good SDKs; token model fits; slightly less flexibility if you later **self-host** for cost/control. |
| **Custom WebRTC (mesh or DIY SFU)** | Weak for v1 | Maximum control, **maximum** engineering and ops cost; easy to ship wrong TURN/STUN, ICE, and reconnection — billing would still need the same contracts below. |

### 2.2 Why LiveKit fits CONNECT

1. **Anonymous sessions:** Access is gated by **short-lived JWTs** the **burner-link-server** issues; claims can include **`roomName`** = opaque id (aligned with **`callSessionId`** or a derived **`livekit_room`**), without exposing wallet or **`deviceId`** to the peer in the token payload visible to the other client (keep claims minimal).
2. **1:1 voice:** Single **Room** with two **Participants** is the standard pattern; mute video by policy (audio-only publish).
3. **Billing alignment:** **Billed duration** can be measured **client-side** (stopwatch from “media connected” → “call ended”) with **one settle** at end, and optionally **cross-checked** later with LiveKit **room/participant** analytics or webhooks (Phase Call-Arch-2+).
4. **Reserve/settle unchanged:** Wallet logic stays in **SQLite** + existing routes; LiveKit does not replace **`call-charge/start`** / **`settle`**.

**Decision:** Adopt **LiveKit** for the first real voice stack unless a hard requirement appears for **PSTN** (then revisit **Twilio** for telephony only, still keeping WebRTC app calls on LiveKit if desired).

## 3. Server contract (what must exist)

### 3.1 Already implemented (billing)

| Endpoint | Role |
|----------|------|
| **`POST /v2/billing/call-charge/start`** | Reserve coins from **`estimatedBillableSeconds`** × voice tariff; idempotency **`call:<callSessionId>:hold`**. |
| **`POST /v2/billing/call-charge/settle`** | Release reserve + final **`call_debit`**; idempotency **`call:<callSessionId>:settle`**. |
| **`GET /v2/billing/wallet`** | Read spendable balance after operations. |

Tariff: **`CONNECT_CALL_TARIFF_JSON`** — for voice-first shipping, configure **voice** rate; **video** can remain for future.

### 3.2 Media token (Phase Call-Arch-2 — implemented)

| Piece | Purpose |
|-------|---------|
| **`POST /v2/calls/livekit-token`** | **Implemented:** **`deviceId` + room link + active room + ≥2 members** → **`token`**, **`url`**, **`roomName`**, **`expiresAt`**. See **`docs/connect-livekit-token.md`**. |
| **Room / session ownership** | **`callSessionId`**: generated by **caller** (UUID); **both** peers agree on the same id for the call attempt (signaled via existing **chat** or **room** payload — not via public profile). Server may map **`callSessionId` → roomId** for authorization: only linked devices for that CONNECT room may request tokens. |
| **Disconnect / end** | **No separate billing webhook required for v1** if **`billedSeconds`** is produced by the **client** and **`settle`** is idempotent. Optional later: **LiveKit webhook** → server logs for fraud/abuse detection (does not replace settle). |
| **`billedSeconds` source of truth (v1)** | **Client stopwatch** between **“both sides connected”** (or **outgoing media established**) and **“call ended”** (user hangup, failure after max retries). Documented in §5. Server **trusts** the integer on settle + idempotency; tighten later with server-side duration if needed. |
| **Retry / idempotency** | **Start:** same **`callSessionId`** + body → duplicate reserve response. **Settle:** same **`callSessionId`** → duplicate settle. **Token:** new JWT per request is OK; authorize **room + device** each time. |

### 3.3 Identity & privacy

- **Chat remains anonymous** between users; **wallet** and **callSessionId** are **not** exposed in room list APIs to the other user except what you intentionally put in **signaling messages** (keep signaling opaque ids only).
- **LiveKit** room names must be **unguessable** (include random component); do not use raw **`roomId`** from URL if it’s short.

## 4. Mobile call flow (target behavior)

**Files (mobile repo, not in this server):** `voiceVideoMediaSession.ts`, `mediaTransportGate.ts`, call UI/state — wire **after** server token route exists.

| Phase | Steps |
|-------|--------|
| **Pre-call** | User taps call → client generates **`callSessionId`** → **`GET /v2/billing/wallet`** (optional UX) → **`POST /v2/billing/call-charge/start`** with **`callType: "voice"`** and realistic **`estimatedBillableSeconds`**. On **402**, show insufficient coins; **do not** open media. |
| **Open media** | Request **`POST /v2/calls/livekit-token`** (with **`deviceId`**, **`roomId`** or invite proof, **`callSessionId`**) → receive token + URL → **LiveKit SDK** connects **audio-only**. |
| **Active call** | UI shows duration; maintain **monotonic timer** for billable window per §5. |
| **End / disconnect / failure** | Always run **`POST /v2/billing/call-charge/settle`** with **`billedSeconds`**, **`reservedAmount`** from start response, **`callType: "voice"`**. Use **try/finally** so settle runs on hangup, background, and many failure paths; **retry settle** on network error until **200** with **`duplicate: true`**. |
| **Insufficient funds mid-call** | Product choice: **soft cap** (warn + auto-end) or **hard stop** when estimated remaining time hits zero. Server does not stream “balance” to peer; optional periodic **wallet** poll on device only. |
| **Peer unavailable** | If signaling fails or peer never joins: **settle with `billedSeconds: 0`** (existing flow supports **0** final debit with settle marker) **after** aborting; **release** reserve via **`reservedAmount`** from start. |

## 5. What counts as billable (`billedSeconds`)

| Question | v1 policy |
|----------|-----------|
| **When does the clock start?** | When **outgoing audio is publishing** and **peer has joined the LiveKit room** (or **first remote audio received** — pick one product rule and stick to it). **Do not** start at ring time. |
| **When does it stop?** | When user ends call **or** terminal failure (see below); stop timer **before** async settle. |
| **Failed connect (never connected)** | **`billedSeconds = 0`**; still **settle** (idempotent) so reserve is **released** and ledger shows **0** final debit where applicable. |
| **Reconnect** | **Same `callSessionId`**, **same** LiveKit room if possible; **timer continues** across short reconnects if the **call** is considered the same session. If you treat reconnect as **new call**, use a **new `callSessionId`** (new reserve/start). **Recommendation:** one **`callSessionId`** per user-visible call; **pause timer** only if you explicitly pause billing (not required in v1 — simplest is **continuous timer** while session active). |

**Future hardening:** compare client **`billedSeconds`** to LiveKit **participant duration** server-side; flag large deltas.

---

## 6. Billing trigger points (summary)

| Event | Billing action |
|-------|----------------|
| User initiates call (paid path) | **`call-charge/start`** (reserve hold). |
| User completes call | **`call-charge/settle`** with measured **`billedSeconds`** + **`reservedAmount`**. |
| Call fails / never connects | **`call-charge/settle`** with **`billedSeconds: 0`**, same **`reservedAmount`** from start. |
| Retry after network error | **Same settle body** → **`duplicate: true`**, no double charge. |

## 7. Exact next implementation phase

**Phase Call-Arch-2 (recommended prompt)**

1. **Server:** Add **`POST /v2/calls/livekit-token`** (or under **`/v2/billing/`** if you prefer) — validate **`deviceId`** + **room membership**, mint **LiveKit** JWT using env **`LIVEKIT_API_KEY`**, **`LIVEKIT_API_SECRET`**, **`LIVEKIT_URL`**; document in **`docs/connect-server-environment.md`**.

**Phase Call-Arch-3+:** Video track, server webhooks for analytics, anti-abuse, optional server-side duration verification.
